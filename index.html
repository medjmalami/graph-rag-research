The RAG Landscape

Four Paradigms, One Goal

Every RAG system grounds LLM output in retrieved knowledge. What separates them is how they retrieve — and what structures they assume about data.

Naive

Basic RAG

Embed → ANN search → generate. Flat chunks, cosine similarity. Works; scales to simple Q&A.

$0.001 / query
~44% factual accuracy
Advanced

Enhanced RAG

Adds query rewriting, hybrid retrieval (BM25 + dense), reranking. Same backbone, better plumbing.

$0.002–0.005 / query
~63% factual accuracy
Agentic

Agentic RAG

LLM-as-controller. ReAct/CRAG/Self-RAG loops. Self-correcting, tool-using, multi-hop capable.

$0.005–0.01 / query
+8.3 F1 on HotpotQA
Graph

GraphRAG

Knowledge graph replaces flat index. Graph traversal + community summaries. Understands relationships.

$0.01–0.05 / query
72–83% on global queries
Key insight: No single paradigm wins everywhere. The 2025–2026 production consensus is hybrid routing — simple queries → vector RAG, multi-hop/thematic → GraphRAG.

Paradigm 2

Advanced RAG

Same vector-search backbone, but optimized at every stage — raising factual accuracy from ~44% to ~63%.

Pre-Retrieval
  • Query rewriting
  • Query decomposition
  • HyDE (Hypothetical Document Embedding)
  • Metadata filtering
Retrieval-Time
  • Hybrid search: dense (vector) + sparse (BM25)
  • Ensemble retrieval
  • Multi-index routing
Post-Retrieval
  • Cross-encoder reranking (+100–300ms)
  • Contextual compression
  • Iterative / multi-hop retrieval passes
Ceiling: Advanced RAG is still chunk-based. It cannot answer thematic queries across 5,000 documents ("What are the main themes?"), and multi-hop retrieval remains heuristic and brittle. This is where GraphRAG becomes necessary.
Best use: Most production RAG applications — customer support, document Q&A, enterprise search. The recommended starting point before considering graph-based approaches.

Paradigm 4 — The Focus

What Is GraphRAG?

Standard RAG treats knowledge as a bag of chunks — semantically related by embedding distance, but structurally disconnected.

GraphRAG replaces the flat index with a structured knowledge graph: entities become nodes, relationships become edges.

Vector RAG FAILS on this: "How is Company A connected to the risk flagged in Document B, and which executives overlap?"
GraphRAG ANSWERS this: "What are the primary operational risks identified across all target companies?"

Architecture 1

Microsoft GraphRAG

Edge et al., arXiv:2404.16130 (Apr 2024) — Open-sourced Jul 2024. 29,800+ GitHub stars by Dec 2024. The reference implementation.

The only architecture that performs pre-computed hierarchical community summarization using the Leiden algorithm. Uniquely capable of answering global thematic queries — and uniquely expensive to index.

Signature Properties

  • Leiden community detection (better connected clusters than Louvain)
  • Multi-level hierarchy: leaf → mid → root community summaries
  • Community reports act as pre-computed answers to thematic queries
  • Three query modes: Local, Global, DRIFT
  • Storage: graph store (NetworkX/Parquet/Neo4j) + vector store + relational tables

When to use

  • Large corpora needing global thematic queries
  • Budget allows full LLM-heavy indexing
  • Stable corpus (expensive to rebuild)

Retrieval Flow Deep Dive · System 1

Microsoft GraphRAG — Query-Time Pipeline

Source note: The original Edge et al. (2024) paper describes only global search. Local search was added in the implementation and is documented in official docs (microsoft.github.io/graphrag), not the paper. Many blog posts incorrectly attribute both modes to the paper.
LOCAL SEARCH entity-centric, specific questions
1Embed Query
Query string is embedded using text-embedding-3-small. This produces the initial retrieval vector. No keyword extraction, no LLM call at this stage.
VECTORFirst signal
2ANN Search — Entity Description Embeddings
Vector search targets the entity description embedding store, not the chunk embedding store. Default top-k = 10 entities. This is the fundamental difference from naive RAG: the ANN index is over entity descriptions, not raw text chunks.
VECTOREntity space, not chunk space
3Graph Expansion — 4-Way Mapping
Retrieved entities seed four parallel graph lookups: (a) entity → text unit mappings → candidate chunks; (b) entity → community report mappings → candidate summaries; (c) entity → related entity mappings → neighbor entities; (d) entity → covariate mappings → structured claims. All are graph lookups, not vector searches.
GRAPH TRAVERSALNo vector search in this step
4Token-Budget Ranking + Filtering
All candidates from step 3 are ranked by a priority score and iteratively added to the context window until the token limit is reached. The exact scoring formula (similarity score only, or also graph degree) is not explicitly specified in source.
PRIORITY RANKING
5LLM Generates Answer
Assembled context (entities + relationships + chunks + community summaries) passed to LLM for synthesis.
LLMFinal answer
GLOBAL SEARCH thematic, corpus-wide questions
1No Retrieval Signal — Broadcast All Communities
There is no vector search in global search. All community summaries at a chosen hierarchy level are retrieved unconditionally. The paper is explicit: community summaries are shuffled and partitioned into chunks of pre-specified token size. No entity lookup, no ANN search.
NO VECTOR SEARCHNo entity lookup
2Map Phase — LLM Scores Each Community
LLM processes each community summary chunk in parallel. For each, it generates a partial answer AND a helpfulness score (0–100). Chunks scoring 0 are filtered out. This is the only "selection" mechanism — done by the LLM, not by vector similarity.
LLM × N communitiesPost-hoc scoring
3Reduce Phase — Aggregate Top-Scored Answers
Intermediate answers sorted by score descending. LLM iteratively adds them to a new context window until the token limit is reached. Final LLM call synthesizes all top-scored partial answers into a single global answer.
LLMFinal answer
Note: Chunks are not retrieved at all in global search. The unit of context is community summaries, not source text. Community report embeddings exist in the implementation but are not used for pre-filtering in the original paper's global search algorithm.
Verdict
Local — first retrieval signalVector search on entity embeddings
Global — first retrieval signalNone — all communities broadcast
ModeEntity-first (local) / Community-broadcast (global)
Chunks accessed viaGraph traversal from entity seeds
ConfidenceHigh — primary paper + official docs

Architecture 2

LightRAG

Guo et al., arXiv:2410.05779 (Oct 2024) — EMNLP 2025. Lightweight alternative that skips expensive community clustering entirely.

Key Design Decision

Instead of pre-computing community summaries, LightRAG stores (entity, relationship) pairs as both graph elements and key-value entries queryable by vector similarity — enabling dual-level retrieval without Leiden overhead.

Low-Level

Entity-specific retrieval — precise, local facts

High-Level

Concept-level retrieval — thematic, abstract queries

Performance vs MS GraphRAG

Indexing token cost reduction~60% lower
Query latency reduction~2× faster
Multi-domain accuracy (EMNLP 2025)Beats MS GraphRAG
Global summarization qualityGood (not excellent)
Outperforms NaiveRAG, RQ-RAG, HyDE, and MS GraphRAG across agriculture, CS, legal, and mixed domains. Supports incremental indexing natively.

Retrieval Flow Deep Dive · System 2

LightRAG — Query-Time Pipeline

Source: Guo et al., EMNLP 2025 (arXiv:2410.05779) — full paper, Section 3.2. All pipeline steps are sourced verbatim from the formal algorithm description.
1LLM Keyword Extraction
The first action is an LLM call, not a vector search. The LLM processes the query and extracts two distinct keyword sets: local query keywords k(l) (specific named entities, concrete concepts) and global query keywords k(g) (abstract themes, high-level concepts). This pre-processing step is unique to LightRAG — neither MS GraphRAG local search nor naive RAG do this.
LLM CALLFirst signal — precedes all retrieval
2Dual Parallel Vector Search
Two vector searches run in parallel against two separate indexes:

→ Low-level search: k(l) keywords matched against the entity name/description index. Targets specific entity nodes.

→ High-level search: k(g) keywords matched against the relation global theme key index. Targets relationship edges that carry LLM-generated global theme descriptors (assigned at indexing time).

Neither index contains raw chunks. The entity index stores entity descriptions keyed by entity name. The relation index stores relationship descriptions keyed by LLM-enriched global theme phrases.
VECTOR × 2 (parallel)Entity index + Relation index
31-Hop Graph Neighbor Expansion
For each retrieved entity node v and each retrieved relation edge e, the system collects their one-hop neighbors: Nv (nodes adjacent to retrieved nodes) and Ne (nodes adjacent to retrieved edges). This enriches the candidate set without deep multi-hop traversal. The formal set is: {vi | vi ∈ V ∧ (vi ∈ Nv ∨ vi ∈ Ne)}.
GRAPH TRAVERSAL1-hop only (not deep)
4Context Assembly from KV Store
Chunks are never independently vector-searched. They appear in context because each entity and relation key-value pair stores a "value" containing the relevant chunk excerpts alongside the entity/relation description. Context = entity descriptions + relationship descriptions + their associated chunk excerpts, assembled from the KV structure.
KV LOOKUPChunks as values, not targets
5LLM Generates Answer
Assembled context passed to the generation LLM. The paper states: "Instead of retrieving chunks as in conventional RAG, we concentrate on retrieving entities and relationships. This approach markedly reduces retrieval overhead compared to the community-based traversal method used in GraphRAG."
LLMFinal answer
Blog misconception to correct: Many posts describe LightRAG as routing between low-level (specific queries) and high-level (abstract queries) as if only one runs per query. The paper's Section 3.2 describes both levels running for any given query. The exact routing logic — always-both vs. classifier-based — is not explicitly specified in the paper. The formal description implies parallel execution.
INDEXING STRUCTURES (what exists at query time)
KG — entity nodes + relationship edges (deduplicated)
VectorIdx[1] — entity name/description embeddings (low-level)
VectorIdx[2] — relation global theme key embeddings (high-level)
KV Store — {entity/rel key → (description, chunk_excerpts)}
No community store. No community summaries.

Vector searchDual parallel (entity + relation indexes)
Query entities extracted?Yes — by LLM (k(l) keywords), not embedding-based
How chunks are retrievedKV lookup — chunk excerpts are values paired with entity/rel keys
Graph traversal depth1-hop neighbor expansion only

Architecture 3

Fast GraphRAG

CircleMind AI, 2024 — No published academic paper. Sourced from official GitHub repository and DeepWiki codebase analysis.

PageRank-Powered Retrieval

Replaces Leiden community summaries and vector-only matching with Personalized PageRank traversal over the entity graph. Query entities seed the graph; relevance propagates through topology; chunks are scored by propagation — never directly searched.

Indexing Structures

  • Knowledge graph stored as igraph sparse matrix (entity nodes + relation edges)
  • HNSW vector index over entity description embeddings (HNSWVectorStorage)
  • Entities-to-relationships sparse matrix (E→R)
  • Relationships-to-chunks sparse matrix (R→C)
  • No community detection. No pre-computed community summaries.

Trade-offs

  • Very fast indexing — no LLM summarization overhead beyond entity extraction
  • Incremental updates supported natively — designed for dynamic corpora
  • PageRank gives graph-topology-aware scoring: connected entities rank higher
  • Interpretable: scoring path entities → relationships → chunks is fully traceable
  • No global summarization capability (no community hierarchy)
  • Less battle-tested at large scale vs LightRAG / MS GraphRAG
  • No peer-reviewed paper — architecture sourced from codebase
Best for: Production systems with frequently changing corpora where rebuild costs are prohibitive, and queries are entity-centric rather than thematic.

Retrieval Flow Deep Dive · System 3

Fast GraphRAG — Query-Time Pipeline

Source: CircleMind official GitHub repository + DeepWiki codebase analysis (query pipeline page). No academic paper exists. Confidence: High for architecture; Medium for precise internal hyperparameters.
1LLM Entity Extraction — Named + Generic
The first action is an LLM call via extract_entities_from_query(). The LLM identifies two distinct entity categories using structured Pydantic output:

→ Named entities: specific proper nouns explicitly mentioned (e.g., "Scrooge", "OpenAI")

→ Generic entities: general concepts or types (e.g., "person", "character", "company")

This dual taxonomy feeds two different HNSW search strategies in the next step.
LLM CALLFirst signal — Pydantic structured output
2HNSW Vector Search — Differentiated by Entity Type
Query string, named entities, and generic entities are all converted to embeddings. Three HNSW searches run with different top-k and threshold settings:

→ Named entities: top_k=1, threshold=0.7 (tight — exact match intended)

→ Generic entities: top_k=20, threshold=0.5 (broad — concept matching)

→ Raw query: top_k=20, threshold=0.5 (broad fallback)

All searches are against the entity description embedding index — not a chunk index.
HNSW × 3 strategiesEntity index only
3PageRank — Graph Structure Scoring
Vector similarity scores from step 2 are converted into a sparse matrix and passed to graph_storage.score_nodes(), which applies Personalized PageRank.

PageRank re-weights the initial vector scores using graph topology: highly connected entities that are also similar to the query receive boosted scores. Entities with many relevant relationships outrank isolated entities with the same similarity score.

This is the core differentiator of Fast GraphRAG — topology-aware relevance scoring on top of vector similarity.
PAGERANKTopology-weighted entity scores
4Score Propagation — E→R→C Matrix Multiplication
PageRank entity scores propagate through two sparse matrices:

→ Entity → Relationship: relationship_scores = entity_scores × E→R_matrix. Relationships that connect high-scoring entities receive high scores.

→ Relationship → Chunk: chunk_scores = relationship_scores × R→C_matrix. Chunks that contain high-scoring relationships receive high scores.

Chunks are never directly vector-searched. Their scores are derived entirely by propagation from entity scores through graph structure.
MATRIX PROPAGATIONE→R→C chain
5Token-Budget Truncation + Context Assembly
Top-scored entities, relationships, and chunks are selected until the token budget is reached. All three are included in the final context — the assembly combines entity descriptions, relationship descriptions, and chunk text for a complete multi-level context.
SCORE RANKINGToken-budget cutoff
6LLM Generates Answer
Assembled multi-level context passed to LLM. The reasoning path is fully traceable: entities → scored relationships → scored chunks.
LLMFinal answer
INDEXING STRUCTURES (what exists at query time)
KG — entity nodes + relation edges (igraph sparse matrix)
HNSW — entity description embeddings (HNSWVectorStorage)
E→R matrix — entities-to-relationships sparse matrix
R→C matrix — relationships-to-chunks sparse matrix
Chunk store — raw chunk text (scored indirectly)
No community detection. No community summaries.
How PageRank differs from naive vector search: In naive RAG, the most similar chunk wins. In Fast GraphRAG, the most similar and most connected entity wins — an entity with 20 relevant relationships scores higher than an equally similar entity with 2 relationships. This is topology-aware retrieval.
Unresolved — not explicitly specified in source: Exact PageRank damping factor and convergence criteria used at query time. The codebase docs describe the mechanism but don't surface these hyperparameters.

Vector searchHNSW on entity embeddings — 3 strategies
PageRank roleRe-weights vector scores by graph topology — not a standalone retrieval step
How chunks are retrievedScore propagation: E→R→C matrix multiplication
Direct chunk vector search?Never
Communities used?None — no community layer exists
ModeEntity-first (vector + graph topology) → propagation to chunks
ConfidenceHigh (architecture) / Medium (hyperparameters)

The 5 things to remember

1

All three GraphRAG systems are entity-first — vector search targets entity embeddings, chunks are always reached downstream through graph structure.

2

MS GraphRAG global search does no retrieval at all — all community summaries are broadcast to the LLM and scored post-hoc.

3

GraphRAG excels on global and multi-hop queries, but is slower and more expensive for simple factoid lookups than vector RAG.

4

The $33K indexing barrier is effectively solved — LazyGraphRAG, LightRAG, and Fast GraphRAG bring it to $33–$100.

5

Hybrid routing is the 2025–2026 production answer — route by query type, not one-size-fits-all.
